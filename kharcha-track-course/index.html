What Is Kharcha Track?

Your expense tracker — and the code behind it

Kharcha Track is a personal expense tracker with superpowers. "Kharcha" means "expense" in Nepali. It does what any good tracker does — records what you spend — but it also predicts your future spending and detects unusual purchases automatically.

Think of it as a financial diary that reads itself back to you and warns you about unusual pages. By the end of this course, you will understand every line of code that makes it work.

The Technology Behind It

Four main technologies work together to make Kharcha Track run. Each handles a different job.

⚙️
Laravel 12

The engine that powers everything — handles requests, talks to the database, enforces rules.

⚡
Livewire 4

Makes pages interactive without writing JavaScript separately — like magic HTML that updates itself.

🎨
MaryUI + Tailwind CSS

The visual layer — beautiful buttons, tables, and forms out of the box.

🗄️
PostgreSQL

The filing cabinet — stores every expense, category, and user permanently.

All of these are built on PHP, a server-side language, and follow a framework pattern that keeps code organized.

Every app has characters. In Kharcha Track, the six database models are like actors in a theater company — each with a specific role to play.

👤
User — The Person

Has roles (admin or regular user), email, and password. Everything in the app belongs to a user — they are the starting point of all data.

⭐
Expense — The Star

Title, amount, date, category, payment method. This is the main thing being tracked — the lead actor that every other character supports.

📁
Category — The Organizer

Groups expenses into buckets like Food, Transport, and Bills. Each user has their own categories plus shared system categories.

📜

Automatically records every create and update as a timeline entry. You never call it directly — it watches silently in the background.

🔮
Forecast — The Oracle

Uses past spending to predict future months. Stores predictions with confidence scores so you know how reliable each guess is.

🔍
Anomaly — The Detective

Scans expenses for unusual amounts and flags them with scores. If something looks off compared to your normal spending, this model catches it.

Livewire — The Magic Mail Carrier

Livewire lets you build interactive pages without writing a separate JavaScript app. When a user clicks a button, Livewire sends a tiny AJAX request to the server and updates only the parts of the page that changed.

Expenses Livewire Component — save() 
public function save(): void
{
    $data = $this->validate([
        'title' => 'required|string|max:255',
        'description' => 'nullable|string',
        'amount' => 'required|numeric|gt:0',
        'category_id' => 'required|exists:categories,id',
        'date' => 'required|date',
    ]);

    if ($this->editing_id) {
        $expense = Expense::findOrFail($this->editing_id);
        $expense->update($data);
        $this->success('Expense updated successfully');
    } else {
        $data['user_id'] = auth()->id();
        Expense::create($data);
        $this->success('Expense created successfully');
    }
    $this->drawer = false;
}
Plain English

Validate — check that all the form data is valid.

Title is required, max 255 characters.

Amount must be a positive number.

Category must exist in the database.

If we are editing an existing expense, find it and update.

Otherwise, attach the current user and create a new one.

Close the side drawer.

Whenever an expense gets updated...

Automatically create a new history snapshot.

Copy the current state of every fillable field.

Mark it as an "update" so we know what happened.

💡
Why This Matters

This pattern — "when X happens, automatically do Y" — is called an event listener. It is everywhere in software: when you get an email, your phone dings. When a file saves, your backup runs. Kharcha Track uses it so every expense change is automatically journaled without anyone having to remember to log it.

04

The Crystal Ball

Forecasting predicts where your money is going. Anomaly detection catches when something looks wrong. Together, they give your expense tracker a sixth sense.

Two Superpowers

Think of these features like a weather service and a smoke alarm. The forecast looks at recent patterns and makes predictions. The anomaly detector does not predict — it reacts when something seems off.

🔮
Forecasting

Predicts next month’s spending using past patterns. Like a weather forecast — it looks at recent history and extrapolates forward.

🚨
Anomaly Detection

Flags expenses that look unusual compared to your normal spending. Like a smoke alarm — it does not predict fires, it reacts when smoke appears.

How Forecasting Works — Moving Average

The forecast engine uses a technique called a moving average. It looks at your last few months of spending and averages them to predict the next month.

Forecast.php — Core Algorithm 
$windowSize = 3;
$recentMonths = array_slice($monthlyTotals, -$windowSize);
$movingAvg = array_sum($recentMonths) / count($recentMonths);
$predictedAmount = round($movingAvg, 2);
$confidence = min(0.9, 0.6 + (count($monthlyTotals) * 0.02));
Plain English

Look at the last 3 months of spending...

Take the average of those 3 months...

That average becomes next month’s prediction...

The more months of data we have, the more confident we are (up to 90%)

1
Your last 3 months: ₹5,000, ₹6,000, ₹5,500
2
Average: (5000 + 6000 + 5500) / 3 = ₹5,500
3
Prediction for next month: ₹5,500
4
Confidence: 78% (you have 9 months of history)
💡
Why Moving Averages Work

This technique — averaging recent data points to smooth out noise — is called a moving average. Stock traders use it, weather services use it, and now your expense app uses it too. It is simple but surprisingly effective for short-term predictions.

How Anomaly Detection Works

The anomaly detector uses normalization to put all expenses on the same scale, then flags anything that strays too far from normal.

Anomaly.php — Normalization 
$amounts = $expenses->pluck('amount')->toArray();
$minAmount = min($amounts);
$maxAmount = max($amounts);
$amountRange = $maxAmount - $minAmount;

$normalizedAmounts = [];
foreach ($amounts as $amount) {
    $normalizedAmounts[] = $amountRange > 0 ? ($amount - $minAmount) / $amountRange : 0;
}
Plain English

Grab all expense amounts as a list...

Find the smallest and largest amounts...

Calculate the range between them...

Scale every amount to be between 0 and 1...

If the range is zero (all same amount), everything is 0

Anomaly.php — Scoring 
$meanAmount = array_sum($normalizedAmounts) / $numSamples;
$distanceFromMean = abs($amount - $meanAmount);
$score = min(1.0, $distanceFromMean * 3);
Plain English

Find the average of all normalized amounts...

How far is this expense from the average?

The further from average, the more suspicious (max anomaly score: 1.0)

Guards & Gates

Authentication & Authorization — the building security system that controls who gets in and what they can do.

Two Sides of Security

Think of a building security system. The front door lock proves who you are. The key card system inside decides which floors you can visit.

🔐
Authentication

Proving who you are. Login with email + password. Laravel checks your credentials and gives you a session — like getting a wristband at a concert.

🛡
Authorization

Proving what you can do. Admin can manage users; regular users can only manage their own expenses. Like having different access levels on a building key card.

💡
Key Terms

Middleware enforces both of these. A role bundles multiple permissions together so you do not have to manage them one by one.

The Role System — Who Gets Which Keys

A seeder runs once during setup to create all the roles and permissions the app needs.

RolesAndPermissionsSeeder.php 
$permissions = [

Plain English

Define 13 specific permissions — each one is a single action like "view users" or "delete roles".

Create a "user" role — regular users who just need basic access.

Users can only access the dashboard — nothing else.

Create an "admin" role — the boss of the system.

Admins get EVERY permission — full control over everything.

The User Model — Layers of Protection

The User model is the identity card of the application. Every security feature connects back to it.

User.php 
class User extends Authenticatable implements MustVerifyEmail
{
    use HasApiTokens, HasFactory, HasRoles, Notifiable;

    protected $fillable = [
        'name', 'email', 'password',
        'email_verified_at', 'previously_verified',
    ];

    protected $hidden = [
        'password', 'remember_token',
    ];

    protected $casts = [
        'email_verified_at' => 'datetime',
        'password' => 'hashed',
    ];
}
Plain English

This User can log in (Authenticatable) and must verify their email (MustVerifyEmail).

It can generate API tokens, have roles, and receive notifications.

These fields can be set when creating or updating a user.

Password and remember token are NEVER shown in responses — security first.

Email verification date is treated as a datetime, and password is automatically encrypted.

💡
Why 'password' => 'hashed' Matters

Laravel automatically encrypts passwords before storing them. Even if someone stole the entire database, they would see scrambled gibberish instead of actual passwords. This is called hashing — a one-way transformation that is practically impossible to reverse.

Middleware — The Checkpoints

Every request to a protected page passes through a series of checkpoints. If any check fails, the request is stopped before it ever reaches the page.

1
Request arrives

A user navigates to /admin/users

2
auth middleware

Are you logged in? If no → redirect to login page.

3
verified middleware

Is your email verified? If no → redirect to verification page.

4
role:admin middleware

Are you an admin? If no → 403 Forbidden.

5
All checks passed!

Load the users management page.

Middleware Layers at a Glance

auth — Must be logged in
verified — Email must be verified
role:admin — Must have admin role
permission:access dashboard — Must have specific permission
