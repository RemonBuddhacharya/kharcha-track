How the Pieces Talk

Data flow: routes direct traffic, Livewire delivers in real-time, model events create automatic carbon copies.

The Traffic Director — Routes

Think of routes as a postal address book. Every URL has a destination, and middleware acts as the security checkpoints along the way.

routes/web.php 
Route::middleware(['auth', 'verified'])->group(function () {
    Route::middleware('role:admin')->prefix('admin')->name('admin.')->group(function () {
        Route::livewire('/users', 'admin.users.index')->name('admin.users.index');
        Route::livewire('/roles', 'admin.roles.index')->name('admin.roles.index');
        Route::livewire('/permissions', 'admin.permissions.index')->name('permissions.index');
    });

    Route::livewire('/forecast', 'expenses.forecasting')->name('forecast.index');
    Route::livewire('/anomaly', 'expenses.anomalies')->name('anomaly.index');
    Route::livewire('/expenses', 'expenses.index')->name('expenses.index');
Plain English

For any page that needs login AND email verification...

If the user is an admin, add the /admin prefix...

Map /admin/users, /admin/roles, and /admin/permissions to their management components.

Map /forecast to the forecasting component.

Map /anomaly to the anomaly detection component.

Map /expenses to the expense list component.

Livewire — The Magic Mail Carrier

Livewire lets you build interactive pages without writing a separate JavaScript app. When a user clicks a button, Livewire sends a tiny AJAX request to the server and updates only the parts of the page that changed.

Expenses Livewire Component — save() 
public function save(): void
{
    $data = $this->validate([
        'title' => 'required|string|max:255',
        'description' => 'nullable|string',
        'amount' => 'required|numeric|gt:0',
        'category_id' => 'required|exists:categories,id',
        'date' => 'required|date',
    ]);

    if ($this->editing_id) {
        $expense = Expense::findOrFail($this->editing_id);
        $expense->update($data);
        $this->success('Expense updated successfully');
    } else {
        $data['user_id'] = auth()->id();
        Expense::create($data);
        $this->success('Expense created successfully');
    }
    $this->drawer = false;
}
Plain English

Validate — check that all the form data is valid.

Title is required, max 255 characters.

Amount must be a positive number.

Category must exist in the database.

If we are editing an existing expense, find it and update.

Otherwise, attach the current user and create a new one.

Close the side drawer.

The Carbon Copy — Model Events

Every time an expense changes, a model event fires and creates an automatic snapshot. This is the observer pattern in action.

Expense Model — booted() 
static::updated(function ($expense) {
    $expense->histories()->create([
        'user_id' => $expense->user_id,
        'title' => $expense->title,
        'amount' => $expense->amount,
        'date' => $expense->date,
        'category_id' => $expense->category_id,
        'payment_method' => $expense->payment_method,
        'is_recurring' => $expense->is_recurring,
        'action' => 'update',
    ]);
});
Plain English

Whenever an expense gets updated...

Automatically create a new history snapshot.

Copy the current state of every fillable field.

Mark it as an "update" so we know what happened.

💡
Why This Matters

This pattern — "when X happens, automatically do Y" — is called an event listener. It is everywhere in software: when you get an email, your phone dings. When a file saves, your backup runs. Kharcha Track uses it so every expense change is automatically journaled without anyone having to remember to log it.
